# 项目协作规则

## 适用范围

本文件中的“课程研究与 Dify 规则”适用于：

- `chat-coach/` 新聊天教练工作区；
- 聊天课程的内容分析、案例拆解、知识单元设计；
- 新 Dify 应用、知识库、检索和回答提示词；
- 与上述工作有关的交接文档和任务规划。

普通代码维护、字幕处理和工程测试仍按项目原有技术规范执行。

## 项目唯一方向

当前项目研究的是讲师如何沟通，并通过 Dify 复现他的分析和说话方式：

```text
女生说了什么
→ 讲师认为她为什么这样说
→ 这代表什么心理、顾虑、窗口、测试或互动信号
→ 男生采取了什么动作
→ 为什么在这个节点采取该动作
→ 他具体是怎么说的
→ 同一目标下还可以从哪些方向说
→ 对方出现不同反应后，下一步如何继续
```

当前产品不是反制产品，也不是对讲师进行批判、揭露或价值审查的产品。未来的反制内容、博客和视频必须作为另一套项目单独建设，不得混入当前知识库。

## 必须遵守的分析立场

1. 先准确理解课程，再归纳课程。不得先建立外部理论或终极分类，再到课程中寻找对应内容。
2. 课程出现什么就记录什么；讲师如何理解女生、如何判断窗口、如何解释动作，就先完整还原他的内部逻辑。
3. 不得主动加入外部心理学理论、道德评价、风险评价、反制视角或对讲师观点的质疑。
4. 不得因为分析者不同意某种说法，就删除、弱化或改写课程内容。
5. 讲师本人明确提出的限制、适用条件或“不建议模仿”等判断，属于课程内容，应照实记录。
6. 可以区分聊天原话、实际动作、讲师解释和分析者复原，但这种区分是为了防止混写，不是为了反驳讲师。
7. 每个判断都要尽量回到完整上下文：关系阶段、前文、回复速度、信息量、主动程度、已经解决的顾虑和后续行动。
8. 不得只摘录一句“话术”。必须说明这句话为什么在该节点出现、承担什么功能，以及如何进入下一步。

## 主知识库回答目标

面向用户的最终回答应尽量包含：

1. 她为什么这样说；
2. 讲师会如何判断；
3. 在课程体系中还可能对应哪些含义；
4. 判断不同含义需要观察哪些上下文信号；
5. 至少三种可选的说话方向；
6. 每个方向的目的、适用信号和不适用情境；
7. 每个方向至少三条具体回复示例，覆盖不同语气和表达强度；
8. 对方积极回应、继续顾虑、转移话题或降低投入后，各准备多条下一句。

回复方向不应被预先固定。应从课程实际内容中发现，例如拉近、后撤、轻松化解、提供筹码、场景描绘、框架转换、给台阶、表达特殊性或继续推进。

面向客户的内容不得只给“一个方向 + 一句话”，也不得用三个标签分别配一句话来假装具有可选性。可选性必须真实存在：客户应能看懂各方向的差异、何时选择，以及同一方向还能怎样换一种自然说法。

## 标签与双链知识对象

课程知识按 OB（Object）组织。每个 OB 是一个完整聊天场景，不是一句孤立话术。

每个 OB 必须同时维护两条链：

1. `course_chain`：场景 → 女生原话 → 讲师判断 → 课程动作 → 课程说法 → 女生反应 → 课程下一步；
2. `supplement_chain`：同一场景 → 未展开的心理分支或缺口 → 沟通目标 → 多种策略 → 多条说法 → 对方不同反应 → 下一轮动作。

对象使用细粒度多标签，包括但不限于关系阶段、场景、女生信号、课程内心理判断、沟通动作、语气和课程概念。一个链路可以同时拥有多个标签。

标签的作用是标记场景、支持召回和关联，不是直接回答问题。禁止向客户只输出“这是 ASD、黄灯、服从性测试、推拉”等标签；必须解释为什么、如何判断、怎样说以及下一步怎么接。

标签体系和对象格式以 `chat-coach/knowledge-object-schema.md` 为准。

正式知识层统一存放在 `chat-coach/ob-knowledge-base/`：

- 全局标签唯一注册表：`课程场景标签.md`；
- 正式对象目录：`OB总索引.md`；
- 每课独立目录：`courses/<COURSE_ID>/`；
- 每课标准文件：`课程内容.md`、`案例动作链.md`、`OB双链.md`、`补足与确认.md`、`召回问法.md`。

`chat-coach/courses/` 是分析工作区，`chat-coach/ob-knowledge-base/` 是正式知识层。未经用户确认的候选内容只能留在分析工作区或正式目录的待确认文件中，不能登记为正式 OB。

`chat-coach/source-material/` 是机械原文资料层，只允许保存由 P02/P03 确定性导出的原文、角色分流、课板 OCR、案例边界和校验报告。该目录中的生成过程不得加入课程分析、标签、OB、补足或回复话术。

每个正式标签和 OB 必须建立双向链接：全局标签的“关联 OB”链接到课程对象；课程对象的标签链接回全局标签锚点。标签 ID 和 OB ID 进入正式知识库后保持稳定，不因标题或文案调整而改变。

检索与问答需要兼容口语近义表达和联想查询，例如“你在干什么”“在做什么”“干嘛呢”“忙什么”“在吗”可能指向相近场景，但最终含义必须结合上下文判断。

## 课程不足与补足机制

项目不仅要复现讲师，也要帮助用户发现课程在“可理解、可检索、可回答、可继续对话”方面没有讲完整的地方。这里的“不足”是课程覆盖或表达完整度的不足，不是从对立立场批判讲师。

可以分析的补足类型包括：

- 讲师给出了判断，但没有解释判断依据；
- 讲师展示了一种回复，但没有给出其他说话方向；
- 讲师给出了话术，但没有说明适用的关系阶段和窗口大小；
- 讲师讲了当前一句，但没有讲对方不同回应后的下一步；
- 同一表达可能有多个课程内动因，但课程只展开了一个；
- 案例包含有效动作链，但课程没有把链路明确总结出来；
- 表达过于依赖具体案例，需要补充相近场景、口语近义说法或可检索表述；
- 课程概念之间的关系没有讲清，需要根据该课内容补成结构。

### 补足的强制规则

1. 每一课单独分析、单独成文，不得把多课补足意见混在一起。
2. 补足内容不得直接写入课程事实、讲师方法或正式 Dify 知识单元。
3. 所有补足默认只是 Codex 提出的候选建议，必须先与用户沟通。
4. 只有用户明确确认后，才能将补足内容标记为已批准，并进入后续知识库设计。
5. 用户未确认、要求修改或否决时，必须保留状态，不得自行采用。
6. 补足应沿着讲师的目标、分析方式和语言风格完善，不得变成另一套思想体系。
7. 补足不能只偏向安抚和退让，还应根据课程场景提供坏坏调侃、轻度占有、主动进攻、框架控制和推拉方向；这些方向不得包含强迫、欺骗或无视明确拒绝。

### 每条补足必须写清

```text
场景：发生在什么关系阶段和上下文
女生原话：她具体说了什么
讲师判断：讲师认为她为什么这样说
讲师处理：课程中具体如何回复或行动
发现的缺口：哪一部分没有讲透或不足以支持 Dify 回答
候选补足：沿着讲师体系，建议增加怎样的分析或说法
补足理由：为什么这样能让判断、回复或下一步更完整
示例：补足后的具体回复和后续分支
来源边界：哪些来自课程，哪些是候选补足
确认状态：待用户确认 / 用户已修改 / 用户已批准 / 用户已否决
```

其中“示例”至少满足：

- 不少于三个明显不同的回复方向；
- 每个方向不少于三条自然语言示例；
- 标明每个方向适合什么上下文和窗口；
- 后续分支不能只有结论，必须提供可直接参考的下一句；
- 示例之间不能只是替换近义词，应体现不同的沟通策略。

每课的候选补足文件使用：

`chat-coach/courses/<COURSE_ID>/supplement-review.md`

## 工作顺序

每门课程必须按以下顺序推进：

1. 完整阅读该课原始材料；
2. 制作开放式内容地图；
3. 还原案例对话和动作链；
4. 提炼讲师实际使用的概念、判断和说法；
5. 形成该课独立的候选补足稿；
6. 向用户展示课程分析与补足建议并等待确认；
7. 用户确认后，才设计该课知识单元和测试问题；
8. 多课验证稳定后，才确定最终 Dify 数据结构、检索流程和提示词。

## 禁止事项

- 禁止把旧 P04、旧 Dify 或预设 ontology 当作新体系答案；
- 禁止为了追求结构完整而编造课程没有出现的判断；
- 禁止把分析者候选补足伪装成讲师原话或课程方法；
- 禁止未经用户确认把补足内容导入 Dify；
- 禁止在主知识库中混入未来反制项目的立场；
- 禁止重复进行没有新增信息的审计迭代。发现方向问题时，应先回到用户目标和课程原文，而不是继续修补旧产物。
