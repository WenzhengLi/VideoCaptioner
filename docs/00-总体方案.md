# 课程视频分析系统总体方案

## 1. 项目目标

输入一段课程、直播、访谈或屏幕录制视频，自动生成一份可追溯的结构化记录：

1. 谁在什么时间说了什么；
2. 同一时间课板、课件或共享屏幕显示了什么；
3. 课板区域即使在左侧、右侧或全屏之间切换，也能持续识别；
4. 将声音、画面和课板截图按统一时间轴合并；
5. 通过本地网页完成上传、预览、人工校正和导出。

第一版核心识别全部在本机运行，不依赖 GPT、DeepSeek 等 LLM，也不需要付费 API。LLM 只作为后续可选层，用于纠错、摘要和知识库整理。

## 2. 第一版范围

### 包含

- 本地视频上传或选择；
- FFmpeg 媒体信息读取和音轨提取；
- 中文语音识别；
- 匿名说话人标记，如 `Speaker 0`、`Speaker 1`；
- 人工将匿名说话人映射为“导师”“助教”等真实名称；
- 自动寻找课板或课件区域；
- 课板在左右位置切换时重新定位；
- 课板换页检测、相似画面去重；
- PaddleOCR 中文识别；
- 时间轴合并；
- JSON、TXT、SRT 和课板图片导出；
- Gradio 本地 Web 界面。

### 暂不包含

- 根据声纹直接推断真实姓名；
- 多人同时讲话的完美分离；
- 潦草手写公式的高精度识别；
- 云端多租户、账号和计费；
- LLM 总结、问答和知识库生成；
- Vue/React 正式前端。

## 3. 总体数据流

```text
输入视频
├─ 音频流水线
│  ├─ FFmpeg 提取 16kHz 单声道 WAV
│  ├─ FunASR 识别文字、时间戳和标点
│  ├─ WeSpeaker 提取声纹并聚类
│  ├─ CAM++ 作为兼容性备用方案
│  └─ 输出 SpeechSegment[]
│
├─ 课板流水线
│  ├─ OpenCV 按间隔抽帧
│  ├─ 镜头/布局变化检测
│  ├─ 候选矩形和文字密度评分
│  ├─ ORB/SIFT 相似特征匹配与位置追踪
│  ├─ SSIM/pHash 换页检测和去重
│  ├─ 选择清晰、少遮挡的代表帧
│  ├─ PaddleOCR 识别文字
│  └─ 输出 BoardSegment[]
│
└─ 时间轴合并
   ├─ 根据时间区间重叠关联声音和课板
   ├─ 保留来源、置信度和截图路径
   └─ 输出 AnalysisResult
```

## 4. 语音与说话人分析

### 4.1 语音识别

第一选择为 FunASR，原因是中文、标点和国内模型下载体验较好。输入统一转换为 16kHz 单声道 WAV，输出至少包含：

```json
{
  "start_ms": 35200,
  "end_ms": 42800,
  "text": "我们先看一下课板上的这个公式",
  "confidence": 0.93
}
```

### 4.2 说话人分离

WeSpeaker 使用 Silero VAD 找到讲话区间，提取 speaker embedding，并通过 UMAP/HDBSCAN 聚类产生匿名人物编号：

```text
Speaker 0：我们先看一下今天的案例。
Speaker 1：好的。
Speaker 0：先从第一段开始。
```

这一步回答的是“是否为同一个声音”，不是“这个人真实叫什么”。第一版在网页提供一次性人工映射：

```text
Speaker 0 → 导师
Speaker 1 → 助教
```

第一版以 WeSpeaker 为主要说话人分离实现，FunASR 集成的 CAM++ 作为安装失败或效果不理想时的备用方案。WeSpeaker 从 GitHub 安装，因为它目前没有标准 PyPI 发行包。

FunASR 与 WeSpeaker 独立处理同一份 WAV：FunASR 输出文字区间，WeSpeaker 输出人物区间，再按照最大时间重叠将人物编号分配给文字片段。这样两套能力可以分别测试、替换和调参。

WeSpeaker 官方元数据主要覆盖 Python 3.8–3.11，第一版统一使用 Python 3.11，避免 Windows 下部分 PyTorch、HDBSCAN、s3prl 和 Whisper 依赖在 Python 3.12 上出现兼容问题。

### 4.3 边界条件

- 对方进入视频、来到家中或之前说过“可以”，都不代表后续行为的持续同意；系统只记录内容，不推断意图。
- 多人同时讲话、背景音乐、变声和远场回声会降低分离准确率。
- 说话人编号在单个任务内稳定；跨视频身份识别属于后续声纹注册功能。

## 5. 课板与课件区域分析

### 5.1 为什么不能固定裁剪

课程视频常见布局包括：

- 左侧课件、右侧人物；
- 右侧课件、左侧人物；
- 课件全屏；
- 实体黑板具有透视角度；
- 主讲人遮挡部分板书。

因此系统不能配置一个永久固定的矩形，而应当在布局变化后重新寻找课板。

### 5.2 首次定位

从整帧生成候选区域，并根据以下指标评分：

- 区域面积；
- 矩形度；
- 边缘和文字框密度；
- 与上一课板的 ORB/SIFT 特征相似度；
- 多帧稳定性；
- 人脸和人体覆盖比例。

第一版允许用户在自动结果错误时手动框选四角，并将修正结果用于后续追踪。

### 5.3 左右换位和持续追踪

课板正常存在时使用特征点或 CSRT 跟踪。出现以下情况时重新扫描整帧：

- 跟踪点数量不足；
- 直方图或感知哈希突变；
- PySceneDetect 判定布局切换；
- 当前区域文字密度突然下降。

重新扫描后，即使课板从左侧移动到右侧，也可根据与上一课板内容的相似特征重新定位。

### 5.4 去重和最佳帧

同一页课板可能持续数分钟。系统按固定间隔抽帧，但只有在 SSIM 或 pHash 差异超过阈值时建立新版本。每个版本保留多张候选帧，按以下指标选代表帧：

- 清晰度；
- 人体遮挡比例；
- 反光比例；
- OCR 文字框完整度。

### 5.5 OCR

首选 PaddleOCR。实体黑板识别前增加透视矫正、CLAHE、二值化和去噪；电子课件通常直接使用原色图像识别。

第一版保留 OCR 原始结果、置信度和人工修订字段，避免错误结果覆盖原始证据。

## 6. 统一领域模型

```text
SpeechSegment
├─ start_ms / end_ms
├─ speaker_id / speaker_name
├─ text
└─ confidence

BoardSegment
├─ start_ms / end_ms
├─ region
├─ image_path
├─ text_lines
└─ confidence

TimelineEntry
├─ start_ms / end_ms
├─ speech[]
└─ boards[]

AnalysisResult
├─ media
├─ speakers
├─ speech_segments
├─ board_segments
└─ timeline
```

所有处理模块只读写这些领域模型，不直接依赖 Web 界面。这样后续可以复用到 CLI、API、批处理或桌面应用。

## 7. 第一版 Web

第一版使用 Gradio，目标是尽快验证整条链路：

1. 上传视频；
2. 选择预计说话人数或自动检测；
3. 启用课板识别；
4. 预览自动检测框；
5. 启动处理并显示阶段进度；
6. 查看说话人文本和课板截图；
7. 修改人物名称和 OCR；
8. 下载结构化结果。

算法验证后，第二版再迁移为 FastAPI + Vue/React，实现后台任务、WebSocket 进度、视频同步时间轴和历史任务管理。

## 9. 最终沉淀

系统最终输出的不是一份不可追溯的总结，而是一组可以重新验证的原始证据：

- 带时间戳和人物标签的讲话；
- 带时间范围的课板截图；
- OCR 原文和人工修订；
- 声音与画面的时间轴关联；
- 可供后续知识库加工的结构化 JSON。

后续知识库或 LLM 只消费这些结构化结果，不参与第一层事实提取，从而减少幻觉并保留可追溯性。
