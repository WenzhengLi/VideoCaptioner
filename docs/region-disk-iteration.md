# 内容区域稳定化与磁盘缓存迭代

## 目标

本轮不是让 AI 帮程序逐帧判断，而是把判断规则固化为可以独立执行的方法：

1. 先定位真正的课板/截图内容区域。
2. 对检测框做抗抖动和留边，再进行图片比较。
3. 图片和 OCR 结果持久化到磁盘，内存只保留少量解码图片。
4. 同一视频的新版本复用历史缓存，但输出到新的版本目录，禁止覆盖旧结果。

## 为什么使用 SQLite，而不是 Docker 数据库

当前任务是单机、单视频、本地顺序读写。SQLite 更适合：

- 无需启动 Docker、数据库服务、端口或账号。
- 元数据查询直接走本地文件，延迟低。
- 数据库可以和视频任务目录一起复制、备份和恢复。
- SQLite 使用 WAL，处理中断后仍可继续恢复。
- 图片不写入数据库 BLOB，仍保存为普通 JPEG，避免数据库膨胀和解码额外开销。

如果以后需要多机器并发处理同一个视频库，可以将相同 Repository 接口替换成 PostgreSQL
或对象存储；当前没有必要增加服务复杂度。

## 磁盘结构

```text
jobs/visual-iterations/
├── _video_cache/<video-fingerprint>/
│   ├── frames.sqlite3
│   ├── frames.sqlite3-wal
│   └── images/
│       └── frame-<frame-index>-<timestamp>.jpg
├── v3-stable-region/
│   ├── artifacts/
│   ├── boards/
│   ├── frames/
│   └── version.json
└── v4-balanced-region/
    ├── artifacts/
    ├── boards/
    ├── frames/
    └── version.json
```

SQLite 每帧保存：

- 时间点和原视频帧序号；
- 图片文件路径；
- 清晰度；
- 是否存在有效文字；
- OCR 分数、文本和行结果；
- 原始内容区域坐标。

OCR 缓存同时按时间点和实际帧序号查找，多个时间点落在同一实际帧时不会重复 OCR。

## 内容区域裁剪与抗抖动

图片差异现在优先比较内容区域，而不是整张视频画面。

1. 使用现有 OpenCV 课板检测器识别左侧、右侧或全屏内容区域。
2. 保存检测器的原始区域，不把一次性的裁剪尺寸当成永久坐标。
3. OCR 裁剪时按区域宽高向四周预留 8%，防止文字贴边被裁掉。
4. 相邻区域 IoU 大于等于 0.5 时，认为是检测框抖动：
   - 对两个框的 `x1/y1/x2/y2` 分别取中位数；
   - 两张图片使用同一组稳定坐标裁剪；
   - 因此 100px 和 110px 的检测框不会因为缩放不同产生虚假变化。
5. 区域 IoU 小于 0.5 时，认为课板可能从左侧移动到右侧或发生布局变化，两张图片分别按
   自己的内容区域裁剪后再比较。
6. 稳定区间选择代表帧时，所有候选帧统一使用该区间的中位数区域计算清晰度和稳定性。

## 内存策略

旧实现的 `FrameObservation` 长时间持有所有 `numpy.ndarray`。新实现默认：

- 图片立即写入共享切片目录；
- 内存 LRU 最多保留 8 张已解码图片；
- 淘汰后需要再次比较时，从 JPEG 文件加载；
- OCR 和帧元数据从 SQLite 恢复；
- 实测峰值 `peak_memory_image_count=8`。

可以通过 `memory_frame_cache_size` 调整内存和磁盘读取之间的权衡。

## 版本结果

旧文件均保留，实验输出包括：

- `output/02.txt`：区域变化敏感、完整度优先版本。
- `output/03.txt`：平衡速度和完整度的实验版本。

| 指标 | 01 旧固定版 | 02 完整度优先 | 03 平衡版 |
| --- | ---: | ---: | ---: |
| 最终课板段 | 366 | 50 | 19 |
| OCR 行数 | 3334 | 852 | 420 |
| 高置信唯一行 | 290 | 247 | 175 |
| 至少出现 2 次内容覆盖 | 100% | 93.6% | 82.1% |
| 至少出现 3 次内容覆盖 | 100% | 97.9% | 88.3% |
| 至少出现 5 次稳定内容覆盖 | 100% | 97.1% | 92.4% |
| 长期稳定内容覆盖 | 100% | 100% | 100% |
| 本轮新增 OCR | - | 33 | 0 |
| 暖缓存总耗时 | - | 211.2 秒 | 17.8 秒 |

冷启动阈值回放：

- 速度优先阈值 `0.35`：预计需要 224 个唯一 OCR，稳定内容覆盖 90.5%。
- 平衡阈值 `0.45`：预计需要 260 个唯一 OCR，稳定内容覆盖 93.3%。
- 敏感阈值 `0.55`：区域内部细微变化会导致超过 500 个唯一 OCR，不再作为默认值。

这些数据衡量的是稳定、重复内容覆盖，不等于全部有效内容的完整度。清洗后三个 TXT 的
复核显示 01 保留了最多独有课板信息，因此程序默认已恢复到 `complete-v1` 固定 5 秒路径。
平衡阈值 `0.45/0.55` 仍由 `adaptive-balanced` Profile 保留，不再作为程序默认值。

磁盘占用实测：

- 共享 SQLite + 原始切片缓存：约 214.5 MB，910 个文件。
- `v4-balanced-region` 最终版本：约 28.4 MB。
- `v3-stable-region` 完整度优先版本：约 2.28 GB。
- 被中断并保留的 `v2-disk-region` 过敏实验：约 24.85 GB；它保留了大量 PaddleOCR
  原图、增强图和调试产物，仅用于审查，不是默认运行结果。

## 无 AI 介入运行

后续只需执行固定脚本：

```powershell
uv run python scripts/rerun_visual_version.py `
  <video> `
  <baseline-analysis.json> `
  <new-version-dir> `
  <new-output.txt> `
  --cache-dir <shared-cache-dir>
```

脚本会自动完成：媒体读取、磁盘缓存恢复、区间拆分、OCR 调度、代表帧选择、课板跟踪、
OCR 文本组合、去重和 TXT/JSON/SRT 导出。最终 TXT 校验成功后，默认删除版本目录中的帧、
OCR 图片和调试产物，仅保留 `version.json` 与外部 TXT；传 `--keep-artifacts` 才保留完整
实验现场。输出目录或 TXT 已存在时会直接报错，禁止覆盖。

完整比较数据：`jobs/visual-iterations/comparison.json`。
